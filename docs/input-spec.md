# 入力仕様書（クオリティ維持のための必須・推奨入力）

生成物のクオリティを一定に保つため、**何を入力として渡すか**を定義する。

---

## 1. 必須入力

| 項目 | 形式 | 説明 |
|------|------|------|
| 企業名 | 文字列 | 正式名称（表記ゆれがある場合は代表表記を1つ）。 |
| 公式情報 | URL または テキスト | **どちらか必須**。URLの場合はホームページ（会社概要・事業紹介・サービス説明が載っているページ）。テキストの場合はそのページの要約・コピー（2000字〜8000字目安）。 |

※ URL のみ渡す場合、実装側でクロール or 手動コピーで「公式情報テキスト」を用意する想定。

---

## 2. 推奨入力（あると提案の具体性が上がる）

| 項目 | 形式 | 説明 |
|------|------|------|
| 業種 | 文字列 | 製造業 / IT・SaaS / 小売 / 金融 など。未入力の場合は生成時に推論させる。 |
| 事業概要 | 1〜3文 | 何をしている会社か。未入力なら公式情報から要約させる。 |
| 注目領域 | リスト（複数可） | 例: 営業 / 開発 / カスタマーサポート / 経理 / 人事。ここを指定すると「活用シーン」をその領域に寄せられる。 |
| 既存のAI・DX | 短文 | 既に導入しているツール・AI利用の有無。重複提案を避け、次の一歩に繋げるため。 |

---

## 3. オプション入力（ディープサーチ等）

| 項目 | 形式 | 説明 |
|------|------|------|
| ディープサーチ結果 | Markdown または プレーンテキスト | 外部検索（Perplexity / 自社検索API / 手動リサーチ）の結果。業界動向・競合の事例・技術トレンドなどを含めると、「同業他社の事例」や「いま導入しやすい技術」を踏まえた案が出しやすい。 |
| 補足メモ | 自由文 | 商談メモ・顧客の課題・予算感など。生成時のトーンや優先度の参考。 |

---

## 4. 入力の受け渡し形式（実装用）

- **単一企業**: JSON または YAML で1ファイル。  
  例: `inputs/company_001.json`
- **複数企業**: 同じ形式のJSONを配列で1ファイル、または1企業1ファイルを `inputs/` に並べる。
- スプレッドシートから読み込む運用にする場合は、「入力用シート」の列名を上記の項目名に合わせ、CSV/JSONに変換してからパイプラインに渡す。

必須項目が欠けている場合は、実行前にエラーまたは確認プロンプトを出し、**公式情報（URL or テキスト）が無い場合はレポート生成をスキップする**ようにする。

---

## 5. 入力 → 出力の対応

- **企業名** → サマリ・活用案の「企業名」列、ファイル名の識別子
- **公式情報（URL/テキスト）** → 事業概要の要約・業種推論・「活用方法」の具体化の主な根拠
- **注目領域** → 活用シーンの優先付け・「活用領域メモ」
- **ディープサーチ結果** → 業界トレンド・事例を踏まえた「活用方法」「期待効果」「備考」
- **既存のAI・DX** → 活用案の「備考」や、提案の優先度づけ

これらを満たすことで、「どの企業にも同じような汎用案」ではなく、**企業ごとに tailorable なレポート**を維持できる。
